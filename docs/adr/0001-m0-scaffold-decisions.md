# ADR 0001 — M0 scaffold decisions (2026-07-17)

- **Initial Alembic migration from ORM metadata.** Context: hand-writing 10 tables risks drift
  from the models. Choice: revision `0001` calls `Base.metadata.create_all`; later migrations
  will be autogenerated diffs. Why: schema and ORM cannot diverge, and the offline `--sql`
  render is covered by a test.
- **Frontend statics via shared volume.** Context: SPEC lists a `frontend` compose service with
  statics served by Caddy. Choice: the frontend image builds `dist/` and copies it into the
  `frontend_dist` volume mounted read-only by Caddy; in dev a `frontend-dev` service runs Vite
  with hot reload. Why: keeps Caddy stock and the build reproducible.
- **First-boot admin from env.** Context: login needs an existing user before M5 user management.
  Choice: on API startup, if the users table is empty and `ADMIN_EMAIL`/`ADMIN_PASSWORD` are set,
  create an active admin (idempotent). Why: simplest bootstrap; the M5 onboarding flow can replace it.
- **In-memory login rate limit.** Context: SPEC §9 asks for rate limiting on login. Choice:
  fixed-window counter per client IP inside the API process. Why: a single API container serves
  ≤20 users; no shared store needed.
- **Custom minimal i18n.** Context: SPEC asks for Italian UI strings via i18n files. Choice: a typed
  `it.ts` constants module re-exported as `t`, no i18n library. Why: one locale today; zero
  dependencies, full type safety; a library can be introduced if a second locale ever arrives.
